# 资讯裁判自检清单

## 使用要求
AI 每次输出“资讯裁判结果”前，必须逐条检查以下问题。只要有一项未通过，就不能直接输出最终结果。

## 强制自检项
1. 是否把公告和小作文混为一谈？
2. 是否把复盘观点当成公司事实？
3. 是否把平台涨停原因当成硬逻辑？
4. 是否遗漏公告、涨停、连板、热榜、小作文中出现的股票？
5. 是否编造股票代码？
6. 是否编造公司业务？
7. 是否把 AI 推理当成证据？
8. 是否没有标注小作文风险？
9. 是否没有标注澄清 / 监管 / 减持风险？
10. 是否给出了买卖建议、目标价或仓位建议？
11. 是否对每条资讯都给出 H / L / C / W / I 与最终评级？
12. 是否对每只股票都给出验证点？
13. 是否对传闻类信息设置降级条件？
14. 是否把 H证据硬度当成最终短线评级？
15. 是否低估了高影响小作文？
16. 是否把小作文当成事实？
17. 是否只判断真实性，没有判断逻辑合理性？
18. 是否只判断真实性，没有判断环境适配度？
19. 是否只判断真实性，没有判断影响力预期？
20. 是否把公告真实性等同于短线影响力？
21. 是否对传闻类信息设置了澄清 / 证伪后的 R 级降级路径？
22. 是否把 W证伪窗口当成了真实性？
23. 是否因为信息不可证伪，就无条件提高评级？
24. 是否检查了不可证伪信息的 L逻辑合理性？
25. 是否检查了不可证伪信息的 C环境适配度？
26. 是否对小作文设置了证伪触发点？
27. 是否识别了公司澄清、监管、问询、董秘互动、权威核查等证伪来源？
28. 是否识别了盘面证伪，例如高开低走、核心股放量滞涨、板块炸板、资金撤退？
29. 是否在 W0 出现后及时降级为 R？
30. 是否把高影响小作文写成了硬事实？
31. 是否把多条不同性质资讯合并成了一行？
32. 是否遗漏了独立催化拆分？
33. 是否把非 A 股主体误写成 A 股股票？
34. 是否遗漏主体类型标注？
35. 是否遗漏“是否A股可交易”判断？
36. 是否在非 A 股主体涉及 A 股时标注了“A股映射链待验证”？
37. 是否把“消息 / 据悉 / 预计 / 宣布”直接升格成公告 / H5？
38. 是否把“主攻支撑 / 弹性支撑”写成了交易结论？
39. 是否在没有资金和盘面事实的情况下输出了主攻池 / 候选池？
40. 是否把资讯侧强度当成最终交易优先级？
41. 是否读取 Markdown 文档级元数据？
42. 是否读取每条资讯 YAML 元数据？
43. 是否记录 `source_time`？
44. 是否记录 `source_time_precision`？
45. 是否记录 `ingest_time`？
46. 是否记录 `analysis_time`？
47. 是否记录 `timezone`？
48. `source_time` 缺失时，是否写 `null`？
49. `source_time_precision` 缺失时，是否写 `unknown`？
50. `ingest_time` 缺失时，是否写 `null`？
51. 是否在单条资讯缺少 `ingest_time` 时继承 `document_ingest_time`？
52. 是否根据 `ingest_time` 优先判断时段属性？
53. 是否误用 `source_time` 替代系统实际获取时间？
54. 是否识别延迟获取？
55. 是否识别旧闻重抓？
56. 是否识别已部分兑现？
57. 是否因为旧闻重抓仍给出过高 I 影响预期？
58. 是否让时间改变了 H 证据硬度？
59. 是否在没有时间依据时编造盘前、盘中、盘后？

## 建议补充检查
- 是否把“中标候选人”误写成“正式中标”？
- 是否把“拟合作”“框架协议”误写成“已经落地订单”？
- 是否把旧闻重炒当成新催化？
- 是否把截图内容直接当成一手纪要？
- 是否漏掉了对“未提供代码”的明确标注？
- 是否对冲突信息做了降级或排雷？
- 是否把同一条资讯里的事实和推演混在一起？
- 是否把海外公司、未上市公司、产品模型、政策机构都混称为“股票”？
- 是否把“7月2日消息”硬补成精确到分钟的时间？

## 输出前最小通过标准
- 每条资讯都有来源类型。
- 每条资讯都有 `source_time / source_time_precision / ingest_time / analysis_time / timezone`。
- 每条资讯都有时段属性和时效状态。
- 每条资讯都有 H / L / C / W / I 与最终评级。
- 每条资讯都有信息性质。
- 每条资讯都有验证点。
- 每条资讯都有证伪触发点。
- 每条资讯都有处理结果。
- 每个被提及主体都进入主体映射表。
- 每条高风险传闻都有降级条件或排雷说明。

## 简版复核口令
输出前可按以下口令自查一遍：

“先读文档级元数据，再读单条 YAML；先看 source_time、precision、ingest_time、analysis_time、timezone；再分来源，再分事实与观点；先看 H，再看 L / C / W / I；不要把 H 和 W 当最终评级；不编代码，不编业务，不给建议；有冲突先降级，有澄清或盘面证伪先排雷。”
