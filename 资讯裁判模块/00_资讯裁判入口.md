# 资讯裁判模块入口

## 模块目标
本模块用于处理 A 股短线投研中的资讯类内容，统一对公告、新闻、政策、研报、纪要、复盘、小作文、传闻、自媒体、图片识别文字、AI 摘要等信息进行裁判、分级、降级与输出。

## 核心原则
1. 资讯不能直接等于事实，必须先裁判再使用。
2. 必须区分事实、观点、推演、传闻、情绪、风险。
3. 必须区分证据硬度、逻辑合理性、环境适配度、证伪窗口、影响力预期与最终短线评级。
4. 小作文不能当事实，但可作为资金攻击线索观察。
5. 公告、监管文件、业绩预告、中标订单、澄清公告属于强事实。
6. 复盘、自媒体、盘前纪要主要用于识别市场共识，不得直接当作公司事实。
7. AI 总结不是证据，只能作为辅助整理。
8. 图片识别文字默认低置信，除非有明确来源。
9. 与公司公告或监管信息冲突的传闻，必须降级处理。
10. 所有资讯最终统一输出为“资讯裁判表”。
11. 真实性低不等于影响力低。
12. 真实性高不等于短线影响力高。
13. 公告可以提高证据硬度，但不必然提高短线影响力。
14. 最终评级 SSS / SS / S / A / B / C / R 表示短线博弈影响力，不表示事实真实性。
15. 信息的短线影响力，不只取决于真假，也取决于它在多长时间内不会被证伪。
16. 证伪窗口只提高博弈空间，不提高真实性。
17. 不可证伪不等于有价值，必须同时评估 L 逻辑合理性和 C 环境适配度。
18. 一旦出现澄清、监管、公告冲突、权威核查或盘面快速否定，证伪窗口关闭，最终评级必须快速降级。
19. 时间不会改变 H 证据硬度，只影响 C 环境适配、W 证伪窗口、I 影响力预期、时段属性、时效状态和处理结果。

## Markdown 时间元数据规范

### 统一格式
统一使用：

```text
ISO 8601 时间格式 + UTC+08 时区
默认时区：Asia/Shanghai
```

A 股场景统一使用北京时间，例如：

```text
2026-07-03T09:35:12+08:00
```

### 标准时间字段
每条资讯统一使用以下字段：

```yaml
source_time: 2026-07-03T09:12:00+08:00
source_time_precision: minute
ingest_time: 2026-07-03T09:14:31+08:00
analysis_time: null
timezone: Asia/Shanghai
```

字段含义：

- `source_time`：资讯原始发布时间。
- `source_time_precision`：原始发布时间精度。
- `ingest_time`：系统获得 / 抓取 / 导入资讯的时间。
- `analysis_time`：AI 实际裁判资讯的时间。
- `timezone`：统一时区。

要求：

1. `source_time` 来自原文、公告、新闻或数据源自带时间。
2. `ingest_time` 来自采集端实际获取时间。
3. `analysis_time` 来自分析端实际执行时间。
4. 正式运行中，短线判断优先使用 `ingest_time`。
5. `analysis_time` 主要用于追溯，不直接决定短线影响力。
6. `source_time` 缺失时不得猜测，写 `null`。
7. `ingest_time` 缺失时写 `null`，并提示采集端正式运行必须补齐。
8. `analysis_time` 缺失时写 `null`。

### source_time_precision 枚举
固定使用以下枚举：

```text
second
minute
hour
date
daypart_morning
daypart_noon
daypart_afternoon
daypart_night
range
unknown
```

边界要求：

1. 原文只有日期时，只保留日期，不得强行补小时分钟。
2. 原文没有发布时间时，`source_time` 写 `null`，`source_time_precision` 写 `unknown`。
3. “7月2日消息”应写为 `source_time: 2026-07-02` 且 `source_time_precision: date`。
4. “7月2日晚”应写为 `source_time: 2026-07-02` 且 `source_time_precision: daypart_night`。
5. 不允许把自然语言时间硬改成伪精确时间。

### Markdown 单条资讯标准格式

````markdown
### 资讯 001：宇树科技 IPO 注册

```yaml
source_time: 2026-07-02
source_time_precision: date
ingest_time: 2026-07-03T09:14:31+08:00
analysis_time: null
timezone: Asia/Shanghai
source_name: 韭研公社APP
source_url: null
collector: manual
```

7月2日，证监会同意宇树科技科创板 IPO 注册申请……
````

要求：

1. 每条资讯建议以 `### 资讯 001：标题` 开头。
2. YAML 元数据块必须紧跟标题。
3. 原文正文放在 YAML 元数据块之后。
4. 多条资讯按 `资讯 001 / 资讯 002 / 资讯 003` 顺序编号。
5. 标题只用于识别，不参与事实判断。
6. AI 分析时必须优先读取 YAML 元数据。

### Markdown 多条资讯文档标准格式

````markdown
# 2026-07-03 资讯采集

```yaml
document_id: jy-20260703-001
document_source: 韭研公社APP
document_ingest_time: 2026-07-03T09:30:00+08:00
timezone: Asia/Shanghai
collector: manual
```
````

要求：

1. 文档级元数据可以提供 `document_ingest_time`。
2. 单条资讯缺少 `ingest_time` 时，可继承 `document_ingest_time`。
3. 单条资讯有自己的 `ingest_time` 时，优先使用单条资讯值。
4. 不允许用文档标题日期替代每条资讯的 `source_time`，除非原文明确该资讯发生在该日期。
5. 文档来源不等于每条资讯来源，单条资讯仍需尽量保留 `source_name`。

## 使用顺序
1. 读取 Markdown 文档级元数据。
2. 读取每条资讯 YAML 元数据。
3. 判断 `source_time / source_time_precision / ingest_time / analysis_time / timezone`。
4. 判断时段属性。
5. 判断时效状态。
6. 资讯源分类。
7. 独立催化拆分。
8. 事实 / 观点 / 推演 / 传闻 / 情绪 / 风险拆分。
9. 板块与股票抽取。
10. 主体类型识别。
11. H 证据硬度评级。
12. L 逻辑合理性评级。
13. C 环境适配度评级。
14. W 证伪窗口评级。
15. I 影响力预期评级。
16. 最终评级 SSS / SS / S / A / B / C / R。
17. 输出资讯裁判表。
18. 输出主体映射表。
19. 输出降级 / 排雷表。
20. 漏股、漏资讯、时间字段、证伪风险自检。

## 裁判流程

### 第一步：识别来源
- 公告类
- 新闻类
- 政策类
- 研报纪要类
- 复盘类
- 小作文与传闻类
- 自媒体类
- 数据异动类
- OCR 与 AI 二次整理类

### 第二步：执行多维评级
按照 [02_证据等级规则.md](./02_证据等级规则.md) 执行，统一采用 H / L / C / W / I + 最终评级。

### 第三步：拆分内容结构
每条资讯至少拆成以下部分：
- 可验证事实
- 主体观点
- 市场推演
- 情绪表达
- 潜在风险

## 逐条资讯拆分规则
一条独立催化必须单独成行。

不得为了压缩输出，把不同来源、不同主体、不同证据硬度、不同证伪窗口、不同风险结构的资讯合并成一行。

以下情况必须拆成多行：
- 同一板块下包含多个不同事件。
- 同一方向下同时出现公告、新闻、小作文、复盘观点。
- 同一段文字里同时包含事实、观点、推演、传闻。
- 同一行业链条中同时包含海外公司、A 股公司、未上市公司、政策机构、产品模型等不同主体。
- 同一主题下不同条目的 H / L / C / W / I 明显不同。
- 同一主题下某条为 H5 强事实，另一条为 H1 / H2 弱线索。
- 同一主题下某条已被证伪，另一条仍处于观察状态。

### 第四步：执行降级与排雷
遇到来源不明、图片截图、二手转述、与公告冲突、时间过期、标题党等情况，必须参考 [09_降级与排雷规则.md](./09_降级与排雷规则.md) 进行降级。

### 第五步：统一输出
所有内容最终落到 [08_资讯输出模板.md](./08_资讯输出模板.md) 的“资讯裁判表”中，不允许只保留原文而不裁判。

## 最低落地要求
- 每条资讯必须有来源类型。
- 每条资讯必须有 `source_time / source_time_precision / ingest_time / analysis_time / timezone`。
- 每条资讯必须有时段属性和时效状态。
- 每条资讯必须有 H / L / C / W / I 与最终评级。
- 每条资讯必须标记信息性质。
- 每条资讯必须写明验证点。
- 每条资讯必须给出处理结果。

## 模块定位
本基础版只提供规则文件和模板文件，不接数据库，不做自动打分程序，不替代研究员判断，只为后续标准化录入与人工裁判提供统一框架。
