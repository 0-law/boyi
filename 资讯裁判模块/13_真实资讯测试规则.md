# 真实资讯测试规则

## 目标
本文件用于真实文章、复盘文章、盘中热点汇总、短研类文章、研报摘录、小作文合集的测试分析。

目标不是给交易建议，而是检验“资讯裁判模块”是否能稳定完成：
1. 逐条资讯拆分。
2. 来源分类。
3. 主体类型识别。
4. A股映射链识别。
5. 事实 / 观点 / 推演 / 传闻 / 情绪 / 风险拆分。
6. 时间字段读取与继承。
7. 时段属性与时效状态判断。
8. H / L / C / W / I 评级。
9. 证伪触发点提取。
10. 降级与排雷。
11. 输出统一表格。
12. 输出自检结果。

## Markdown 时间元数据要求
每条资讯必须尽量输出：

```yaml
source_time: null
source_time_precision: unknown
ingest_time: null
analysis_time: null
timezone: Asia/Shanghai
```

如果原文只有“7月2日消息”，则：

```yaml
source_time: 2026-07-02
source_time_precision: date
ingest_time: null
analysis_time: null
timezone: Asia/Shanghai
```

如果采集端提供 `ingest_time`，则优先用 `ingest_time` 判断盘前 / 盘中 / 盘后。

## 时间处理边界
1. 统一使用 `ISO 8601 + UTC+08`，默认时区 `Asia/Shanghai`。
2. 原文只有日期时，只保留日期，不得补具体时分秒。
3. 原文没有发布时间时，`source_time` 写 `null`，`source_time_precision` 写 `unknown`。
4. `ingest_time` 缺失时可以先写 `null`，但正式运行应尽量补齐。
5. `analysis_time` 缺失时写 `null`。
6. 时段属性判断优先级固定为 `ingest_time > source_time > 无法判断`。
7. 时间不会改变 H 证据硬度，只影响 C / W / I、时段属性、时效状态和处理结果。
8. 不允许编造盘前、盘中、盘后。

## 输入要求
输入可以是完整文章，也可以是多条混合资讯。

输入中可能包含：
- 公告
- 新闻
- 政策
- 研报观点
- 纪要
- 复盘信息
- 小作文
- 传闻
- 自媒体观点
- 涨停原因
- 异动原因
- OCR 文本
- AI 摘要
- 个股逻辑
- 板块逻辑
- 海外公司新闻
- 未上市公司新闻
- 产品模型发布
- 时间节点 / 事件日历
- 市场情绪描述

## 逐条资讯拆分规则
一条独立催化必须单独成行。

不得为了压缩输出，把不同来源、不同主体、不同证据硬度、不同证伪窗口、不同风险结构的资讯合并成一行。

以下情况必须拆分：
1. 同一板块下包含多个不同事件。
2. 同一方向下同时出现公告、新闻、小作文、复盘观点。
3. 同一段文字里同时包含事实、观点、推演、传闻。
4. 同一行业链条中同时包含海外公司、A 股公司、未上市公司、政策机构、产品模型等不同主体。
5. 同一主题下不同条目的 H / L / C / W / I 明显不同。
6. 同一主题下某条为 H5 强事实，另一条为 H1 / H2 弱线索。
7. 同一主题下某条已被证伪，另一条仍处于观察状态。

## 主体类型规则
资讯裁判必须区分：
- A股上市公司
- 港股上市公司
- 美股上市公司
- 海外公司
- 未上市公司
- 政策/监管机构
- 行业协会/交易所
- 产品/模型/项目
- 行业方向
- 宏观数据
- 未知主体

非 A 股主体不得直接当作 A 股股票。

如需映射 A 股，必须标注“A股映射链待验证”。

## A股映射链规则
当资讯主体不是 A 股上市公司，但文章试图把它映射到 A 股板块或个股时，必须标记“映射链待验证”。

必须至少回答：
1. 该主体本身是否 A 股可交易？
2. 如果不是，是否有明确 A 股供应链、客户、订单、参股、合作、业务映射？
3. 原文是否给出了 A 股映射标的？
4. 映射链来自公告、新闻、研报、小作文，还是 AI 推演？
5. 是否需要另行验证 A 股映射链？

如果原文没有给出 A 股映射链，不得脑补 A 股个股。

## 来源升级限制
原文只写“消息 / 据悉 / 预计 / 传出 / 媒体称 / 业内人士表示 / 公司宣布 / 有望 / 或将 / 可能 / 计划 / 拟”时，不得直接升为公告或 H5。

只有明确出现以下口径，或可回溯到正式披露文件时，才可标为公告 / H5：
- 公告
- 交易所披露
- 定期报告
- 业绩预告
- 监管文件
- 公司公告原文

## 资讯侧支持边界
“主攻支撑”和“弹性支撑”只表示资讯侧支持，不代表最终交易结论。

在没有接入以下信息前：
- 板块资金流入
- 个股资金流入
- 成交额
- 换手率
- 分时承接
- 涨停数量
- 连板高度
- 龙头地位
- 板块扩散
- 高位股风险
- 亏钱效应

不得把“主攻支撑 / 弹性支撑”解释为交易主攻、买入、加仓、目标价或持仓建议。

## 时间节点 / 事件日历处理规则
时间节点本身通常不是强催化，但会影响短线预期、验证窗口和证伪窗口。

时间节点需要单独成行，不得被并入其他资讯中。

## 分析顺序
必须按以下顺序执行：
1. 读取 Markdown 文档级元数据。
2. 读取每条资讯 YAML 元数据。
3. 判断 `source_time / source_time_precision / ingest_time / analysis_time / timezone`。
4. 判断时段属性。
5. 判断时效状态。
6. 文章整体质量评估。
7. 独立催化拆分。
8. 来源分类。
9. 主体类型识别。
10. A股映射链判断。
11. 事实 / 观点 / 推演 / 传闻 / 情绪 / 风险拆分。
12. H 证据硬度评级。
13. L 逻辑合理性评级。
14. C 环境适配度评级。
15. W 证伪窗口评级。
16. I 影响力预期评级。
17. 最终评级。
18. 证伪触发点提取。
19. 降级风险识别。
20. 输出表格。
21. 输出自检结果。

## 输出结构

### 一、文章质量评估
- 主要信息源
- 是否混合公告 / 新闻 / 小作文 / 复盘观点
- 是否存在来源不明内容
- 是否存在多个独立催化
- 是否需要逐条拆分
- A股上市公司主体
- 非A股主体
- 未上市公司
- 产品 / 模型 / 项目
- 政策 / 监管机构
- 行业方向
- 是否存在过度推演
- 是否存在概念擦边
- 是否存在把传闻当事实
- 是否给出明确 A 股映射
- 是否存在非 A 股主体直接映射 A 股的风险
- 是否具备传播性
- 是否具备不可证伪窗口
- 主要降级点

### 五、自检结果
- 是否遗漏文章中出现的股票 / 主体
- 是否把非 A 股主体误写成 A 股股票
- 是否编造股票代码
- 是否编造公司业务
- 是否把小作文当成事实
- 是否把复盘观点当成公司事实
- 是否把“消息 / 预计 / 据悉 / 宣布”直接升格为公告 / H5
- 是否把多个不同性质资讯合并成一行
- 是否把 H 当成最终评级
- 是否把 W 当成真实性
- 是否因不可证伪而无条件提高评级
- 是否把“主攻支撑 / 弹性支撑”写成交易结论
- 是否给出买卖建议 / 目标价 / 仓位建议
- 是否对每条高风险资讯给出证伪触发点
- 是否读取文档级元数据
- 是否读取每条资讯 YAML 元数据
- 是否完整输出时间字段组

## 测试合格标准
1. 所有独立催化尽量单独成行。
2. 非 A 股主体不得混入 A 股股票。
3. 未提供代码时不得编造代码。
4. 原文未明确公告时，不得直接标为公告 / H5。
5. 小作文不得当事实。
6. 复盘观点不得当公司事实。
7. 每条高风险资讯必须给出证伪触发点。
8. 每个非 A 股主体必须标注主体类型。
9. 每个非 A 股主体必须标注是否 A 股可交易。
10. 每个非 A 股主体若涉及 A 股映射，必须标注“A股映射链待验证”。
11. 不得输出买卖建议、目标价、仓位建议。
12. 不得缺失 `source_time / source_time_precision / ingest_time / analysis_time / timezone`。
